## 目标

你不需要先大改代码；你需要先能“重新掌控”这套多智能体系统，并在学习过程中判断哪些功能应该删掉/合并/简化。

## 交付物（学习优先）

1. 一份`backend/app/agents/ARCHITECTURE_GUIDE.md`：包含请求数据流、每个节点的输入/输出(state字段)、以及必须保持的“不变量”。
2. 一份`backend/tests/EVAL_PLAN.md`：给出质量优先的评估指标与基线运行方式（用现有`backend/tests/test_refactor_quality.py`）。
3. 一份“裁剪候选清单”（写在同一份`ARCHITECTURE_GUIDE.md`或独立`backend/app/agents/PRUNE_CANDIDATES.md`），把每个模块的必要性、复杂度成本、以及是否会提升质量列清楚。

## 关键学习点（本次先用读代码确认）

1. 全链路数据流（必须读懂的闭环）
  - `api`接收`TripRequest`
  - `MapAgentsSystem._build_initial_state()`写入`messages/next/city/.../trip_intent={}`等
  - `GraphBuilder`：`intent_analyzer -> supervisor -> workers(可并发) -> planner_agent -> END`
  - `workers.py`：分别填充`attractions/weather_info/hotels/routes`
  - `Planner.generate()`：从这些字段生成`TripPlan.days[*]`与`weather_info/overall_suggestions/budget`
2. 当前“复杂但效果不增”的高优先级矛盾点（学习过程中重点核对）
  - API 路由方法不一致：`backend/app/api/routes/map_agents_router.py`里调用`system.plan_trip_async`，但`backend/app/agents/main.py`只定义了`plan_trip`（这类问题会让体验/质量波动大、看起来像“改了但没提升”）。
  - Route 逻辑矛盾：`backend/app/agents/prompts/agents.py`里的`ROUTE`提示要求 origin/destination 用“景点名称”，但`backend/app/agents/workers.py`为`route_agent`的 context builder 提供“坐标”，且 MCP 方向工具输入要求坐标（这会导致 route 失败/重试/输出被过滤）。
  - Planner 路线数据清洗错误：`backend/app/agents/workers.py`的`_extract_planner_fields()`只保留`name/address/location/...`等字段白名单，缺少 route 的字段；因此`routes_context`很可能被清洗为空，Planner 无法基于真实路线写入（会逼 Planner 叙事补全，降低 grounding）。
  - 餐饮/避雷 grounding 缺口：系统没有餐饮检索类 worker，但`frontend`与`TripPlan` schema 支持`days[].meals`，planner prompt 又倾向输出具体餐厅/避雷（因此容易胡编或与检索证据脱节）。

## 评估基线（质量优先）

- 使用现有：`backend/tests/test_refactor_quality.py`
- 记录并比较以下指标（输出到`backend/tests/output/`）
  - `parse_and_build_plan`是否成功（异常/兜底触发率）
  - `days[].attractions`数量分布、`hotels/routes/meals`非空率
  - `budget`是否缺失或异常（质量角度：预算“像不像真的”，以及是否依赖不可验证输入）
  - 抽样检查“餐饮/避雷/交通路线”的具体性与是否容易出现明显胡编

## 裁剪决策输出逻辑（学习→裁剪）

- 对每个模块给出三项结论：
  - 保留：明确提升质量或提供关键约束证据
  - 降级：保留但强约束输出为空/通用（例如 meals）
  - 删除/合并：增加复杂度但当前不提升质量（例如 routes 数据清洗/路由 prompt 矛盾链条导致的重试成本）
